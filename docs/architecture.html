Kafka Partition Scaling — Consumer Group Model
voiceops.calls — 6 partitions, key=business_id
Same partitions, three consumer groups, independent offsets
Group: "archiver" (1-2)
Bottleneck: S3 write I/O (~50ms/batch)
P0P1P2P3P4P5
1 Archiver handles all 6
Group: "extractor" (3-6)
Bottleneck: LLM API (~2s/call)
P0→AP1→AP2→BP3→BP4→CP5→C
3 Extractors, 2 partitions each
Group: "observer" (1)
Lightweight metrics collector. Reads all partitions. Writes to pipeline_metrics. Never needs to scale — it's measuring, not computing.
Fargate Autoscaling
Consumer lag > 1000 → scale Extractors 2→6
Consumer lag < 100 → scale back to 2

9am spike → lag grows → more Extractors → lag drains → scale down midnight.

PIPEDA Compliance — Data Flow
Consent
Gate at Audit 1
consent_obtained=false → event rejected before Kafka. No consent, no processing, no storage. Voice system responsible for obtaining consent.
PII Minimization
Bronze/Silver Split
Bronze (S3): full PII — transcript, phone, names. Encrypted, restricted, 90-day retention.
Silver (Postgres): scrubbed — customer_hash only. No raw phone. LLM prompted: "the caller" not real names.
PII Leak Detection
Audit 2b Guardrail
Regex scanner on LLM output: phone patterns, name prefixes (Mr./Mrs.), street addresses. Leak detected → DLQ, not Postgres. Catches what the prompt missed.
Data Sovereignty
LLM in Canada
Production: AWS Bedrock ca-central-1 (Montreal). PII never leaves Canada. Demo: mock extractor (no external calls). Fallback: local model (Ollama/vLLM).
Right to Deletion
customer_hash Lookup
Customer requests deletion → DELETE FROM call_extractions WHERE customer_hash = ?. Bronze S3: rewrite Parquet excluding matching rows. DLQ: purge matching call_ids. Index on customer_hash enables fast lookup.
Encryption + Retention
At Rest + In Transit
S3: SSE-KMS. Postgres: RDS encryption. Kafka: TLS. All connections: SSL/HTTPS.

Retention: Kafka 7d, Bronze 90d→Glacier→delete 1yr, Silver 2yr, DLQ 90d.
